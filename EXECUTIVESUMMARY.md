**Executive Summary: An Implementation of OHDSI OSS on InterSystems IRIS**

**Introduction and Vision**  
The goal of this project is to develop and share a robust, scalable **implementation** that employs the **OMOP Common Data Model** on **InterSystems IRIS Community**. This architecture facilitates the onboarding of the comprehensive **OHDSI Open Source Software (OSS)** suite as containerized services within an **IrisCluster** and built for enterprise deployment or local evaluation out of the gate using Kubernetes workload patterns.

By presenting this at the **2026 OHDSI Symposium**, I aim to offer the community a new, high-performance pathway for generating real-world evidence. I want to be clear: this is not the display of a  commercial product, but an implementation built from the ground up using OHDSI’s open-source tools as a reference architecture. Furthermore, while I am an aspiring contributor to the OHDSI mission, **the responsibility for the maintenance, tool-compatibility, and operational success of this IRIS-specific architecture rests solely with me.** My objective is to provide a proven, reproducible implementation that empowers the community without creating an additional support burden for existing maintainers of the sub projects in use.

**Technical Foundation: IrisCluster and Containerization**  
This architecture utilizes **InterSystems IRIS** as the primary database for the **OMOP CDM**, managing standardized health data and the complex vocabulary tables required for international harmonization. The deployment strategy draws inspiration from the **Broadsea** distribution, which uses container technology to simplify the setup of the OHDSI stack and celebrates recent contributions for IRIS interoperability within the OHDSI tools.

The reference architecture integrates:

• **ATLAS and WebAPI:** Providing a web-based interface for cohort definition, population characterization, and the design of scientific analyses.  
• **HADES:** A collection of R packages for large-scale analytics, causal effect estimation, and patient-level prediction.  
• **Data Quality Dashboard (DQD) and Achilles:** Tools used to quantify data integrity and provide visual characterization of the CDM instance.

**Agentic ETL: Leveraging the ADK and OHDSI Tooling "Hints"**  
A core innovation of this implementation is the employment of **agentic operations** to streamline the complex **Extract, Transform, Load (ETL)** process. 

Traditional OHDSI ETL development relies on a suite of manual tools to design the transition from source data to the CDM:

• **WhiteRabbit:** Scans source data to produce reports on tables, fields, and values.  
• **Rabbit-in-a-Hat:** Uses WhiteRabbit's reports to visually connect source data to the CDM and define transformation logic.  
• **Usagi:** Facilitates the mapping of source codes to the standard terminologies in the OMOP Vocabulary.

In this implementation, I utilize the **Google ADK (Agentic Development Kit)** to “grease” the ETL/Transformation from FHIR Resources.. Rather than replacing the OHDSI tools, the **agentic AI** uses the outputs and "hints" generated by **WhiteRabbit**, **Rabbit-in-a-Hat**, and **Usagi** as foundational logic. By consuming these mapping specifications and scan reports, the AI agent can automate the generation and execution of transformation code, reducing the manual effort typically required to populate an OMOP instance.

**A Commitment to Transparency and Responsibility**  
I recognize that OHDSI is an international community built on the pillars of transparency and reproducibility. To demonstrate my commitment to these values:

• **Compliance Documentation:** I am adopting the **OHDSI-compliance** framework, providing templates for a Record of Processing Activities (ROPA), an Information Security Concept, and an Operational Concept tailored for this IRIS-based stack.

• **Open Source:** All configuration scripts and container definitions for this reference architecture will be shared, ensuring that any organization can replicate the environment independently and included community versions of all software.

• **Personal Accountability:** Because InterSystems IRIS is a novel platform for many OHDSI tools, I am personally responsible for the "heavy lifting" required to ensure seamless interfacing between the IRIS database and tools like **WebAPI** and **HADES**.

**Conclusion**  
By the 2026 Symposium, I intend to show that this implementation of the OHDSI Open Source Solutions as a  high-performance, AI-augmented path for the community, researchers, hackers, and organizations.
